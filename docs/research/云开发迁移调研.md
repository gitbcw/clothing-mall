# 不走 ICP 备案情况下的方案结论与改造清单

## 1. 当前背景

我目前的系统分为三部分：

1. 小程序
2. 后台管理系统
3. 自建后端（目前在服务器上）

当前卡点是：**没有办法完成 ICP 备案**，因此希望评估是否能通过 **微信小程序云开发 / CloudBase / 云托管** 的方式上线，并尽量避免使用自有备案域名。

---

## 2. 总体结论

### 2.1 可以实现的目标

如果目标是：

- **小程序先上线**
- **不使用自有备案域名**
- 后端能力尽量迁移到 **CloudBase 云开发体系**
- 小程序侧尽量通过 **云函数 / callContainer / 云开发能力** 调后端

那么这条路 **总体可行**。  
原因是：小程序通过云开发调用后端时，官方支持 **内网通信**，并且 **无需在小程序后台配置服务器域名**。

---

## 3. 需要明确区分的两件事

### 3.1 “不用自有备案域名” ≠ “完全没有域名”

即使不买自己的域名，系统里仍然可能存在：

- CloudBase 默认域名
- 静态托管默认域名
- 云托管默认域名

但关键点在于：

- **可以不使用自己的备案域名**
- **可以不暴露自己的公网 API 域名**
- 但平台分配的默认域名，官方通常只建议用于**开发测试**，不建议作为长期正式公网入口。

### 3.2 小程序内网调用 和 公网域名访问 不是一回事

#### 小程序内网调用
如果小程序使用 `wx.cloud.callContainer` 或云函数调用云托管/云开发后端：

- 走的是云开发体系内访问
- **无需配置服务器域名**
- 更适合“免备案”上线思路。

#### 公网访问
如果是浏览器、第三方系统、Webhook 去访问你的服务：

- 就会涉及公网访问
- 平台默认域名虽然能用，但官方一般不建议作为正式生产公网入口
- 默认域名可能存在**频率限制 / 性能限制 / 不承诺 SLA**。

---

## 4. 针对我当前项目的结论

## 4.1 小程序部分

### 结论
**建议优先迁到 CloudBase / 云开发体系。**

### 原因
这样做后：

- 小程序可直接调用云函数
- 或通过 `callContainer` 调云托管服务
- 从而避开“必须有自有备案域名”这个问题。

### 要做的事
- 接入云开发环境
- 改造小程序请求方式
- 尽量把原来 `wx.request` 请求自建后端的部分，改成：
  - `wx.cloud.callFunction`
  - `wx.cloud.callContainer`
  - 或直接使用云数据库 / 云存储

---

## 4.2 后端部分

### 结论
如果不走备案，后端最好不要继续维持“自建服务器 + 公网 API 域名”模式。

更可行的两条路：

#### 方案 A：迁到云托管
适合：
- 现有 Java 后端已有 Dockerfile
- 希望少改代码

做法：
- 将现有 Java 服务部署到云托管
- 小程序端通过 `callContainer` 调用
- 尽量不依赖公网默认域名给外部访问

#### 方案 B：逐步改造成云函数
适合：
- API 比较独立
- 能接受重构

做法：
- 把部分后端接口拆成云函数
- 数据写入云数据库
- 文件放云存储

### 对当前项目的建议
由于当前后端是 **Java 且已有 Dockerfile**，短期内更适合先走：

> **Java 后端先迁到云托管，供小程序内网调用；后续再按模块逐步迁到云函数。**

---

## 4.3 微信支付

### 结论
微信支付**不一定必须依赖你自己的备案域名**，前提是走 **CloudBase 官方支付方案**，由云函数或工作流承接支付回调。

### 含义
如果支付是按传统服务器方式接入，则需要 `notify_url` 回调地址；  
但如果使用 CloudBase 官方支付实践，可以由云函数接受支付回调，**无需自建回调服务**。

### 建议
- 若决定走“无备案域名方案”，支付尽量改成 **CloudBase 官方支付接法**
- 不建议继续沿用“自建服务器 notify_url 回调”的旧模式

---

## 4.4 企业微信

### 结论
企业微信这块要分两种情况：

#### 情况 A：我后端主动调企微 API
比如：
- 发消息
- 发小程序卡片
- 定时调用企微接口

这种通常问题不大，本质是**我主动请求外部 API**。云托管/云函数可以出公网访问外部接口。

#### 情况 B：企业微信反过来回调我
如果用了：
- 事件回调
- 消息回调
- Webhook 回调

这就属于“外部系统主动访问我”，此时就更容易碰到公网入口问题。

### 建议
- **能只用主动调用就只用主动调用**
- 尽量避免企微主动回调到我自己的服务

---

## 4.5 后台管理系统

### 结论
后台前端可以做，但这是整个方案里**最容易重新碰到域名问题**的部分。

### 两种思路

#### 方案 A：Vue 后台前端 + CloudBase 静态托管
可行，官方支持 Vue 单页应用部署到静态托管。

但要注意：

- 后台前端访问时，仍然会有一个 Web 域名入口
- 即使不用自己的域名，也至少会用到平台默认托管域名
- 如果前端通过 JS SDK 调用云函数/数据库，需要配置安全域名。

#### 方案 B：直接用 CloudBase 云后台 / 低代码后台
如果后台功能主要是：
- CRUD
- 表单
- 列表
- 简单管理操作

这条路会更省事。

### 最大风险点
如果后台前端是浏览器里直接请求你自己暴露出来的 HTTP API，那么你又会回到：

- 默认公网域名不建议正式生产
- 自定义域名更推荐
- 而自定义域名又通常会涉及备案

### 建议
后台尽量改成：

> **前端静态托管 + JS SDK / 云函数 / 云数据库**

而不是：

> **前端 + 自建公网 REST API**

---

## 5. 如果不走 ICP 备案，我应该做哪些事情

## 第一阶段：确定总体路线

### 建议路线
- **小程序：迁云开发**
- **后端：先迁云托管，再逐步云函数化**
- **支付：优先改成 CloudBase 支付方案**
- **企微：尽量只保留主动调用**
- **后台：优先考虑静态托管 + 云函数 / 云数据库**

---

## 第二阶段：具体改造事项

### 5.1 小程序改造
- 开通 CloudBase 环境
- 初始化云开发 SDK
- 将原来的 `wx.request` 改成：
  - `wx.cloud.callFunction`
  - `wx.cloud.callContainer`
- 能直接使用云数据库的场景，尽量减少中间 API

### 5.2 后端改造
- 现有 Java 服务打包成容器并部署到云托管
- 小程序通过 `callContainer` 调用服务
- 把高频、独立、无状态接口逐步拆到云函数

### 5.3 数据层改造
- 评估现有数据库是否迁移到云数据库
- 如果暂时不迁，可先保留原数据结构，由云托管服务访问
- 长期建议评估哪些数据适合直接放到云开发体系

### 5.4 支付改造
- 重新评估支付接入方式
- 尽量切到 CloudBase 官方支付方案
- 使用云函数/工作流处理支付回调

### 5.5 企业微信改造
- 梳理当前企微能力清单
- 区分：
  - 主动调用型
  - 回调型
- 能取消回调的尽量取消

### 5.6 管理后台改造
- 评估后台是否继续保留现有 Vue 项目
- 若保留：
  - 前端部署到静态托管
  - 数据访问优先走 JS SDK + 云函数
- 避免继续维护一套公网 HTTP API

---

## 6. 推荐的最终架构（无备案域名优先）

### 推荐目标架构

#### 小程序侧
- 小程序
- 调用云函数 / callContainer
- 访问云数据库 / 云存储

#### 后端侧
- Java 后端先放云托管
- 后续逐步将接口迁到云函数
- 第三方 API（企微、物流等）由云函数或云托管主动请求

#### 支付侧
- 使用 CloudBase 官方支付能力
- 支付回调由云函数 / 工作流承接

#### 后台侧
- Vue 后台前端部署到静态托管
- 通过 JS SDK / 云函数 / 云数据库工作
- 尽量不使用自建公网 API

---

## 7. 风险与边界

### 7.1 能免掉的是“自有备案域名”
可以尽量做到：
- 不买自己的域名
- 不做 ICP 备案
- 不暴露自己的公网 API 域名

### 7.2 不能误解成“所有默认域名都适合正式生产公网入口”
平台默认域名通常：
- 可用
- 能测试
- 能短期跑通
- 但官方不建议作为长期正式生产公网入口
- 可能存在频率限制 / 性能限制 / SLA 不承诺。

### 7.3 最容易出问题的不是小程序，而是 Web 后台
小程序内网调用云开发后端，是这套方案最顺的一环；  
真正容易把系统重新拉回“要公网正式入口”的，通常是：

- Web 后台浏览器访问后端
- 外部系统回调
- 自建 HTTP API 暴露

---

## 8. 最终结论

如果我**不走 ICP 备案**，那最合理的路线是：

1. **小程序全面迁向云开发**
2. **后端优先迁云托管，逐步云函数化**
3. **微信支付尽量切到 CloudBase 官方支付方案**
4. **企业微信尽量只保留主动调用**
5. **后台系统尽量改成静态托管 + JS SDK + 云函数 / 云数据库**
6. **避免继续依赖自建公网 API 和自有域名**

### 一句话总结

> **如果目标是不备案上线，那么最关键不是“有没有平台分配域名”，而是“系统是否能尽量运行在 CloudBase 体系内，并避免依赖自建公网 API”。**
